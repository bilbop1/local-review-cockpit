# Clipping Ops Cockpit v0.2.0

Release date: 2026-06-14

## Executive Summary

v0.2.0 moves Clipping Ops from the retired native Swift cockpit to the local web cockpit and turns the MiniMax/Hermes review factory into the default operator workflow. It also adds the approval-to-posting queue: approving a non-demo review kit creates a publish package and schedules it into the next future local `:14` slot.

Fresh installs are deliberately locked. The system can prepare and check packages immediately; it cannot live post until provider key readiness, exact local Upload-Post profile, selected-platform warm-up, live mode, and local auto-post or final GUI confirmation all pass. TikTok is the default Upload-Post platform; Instagram, YouTube, Facebook, and X stay blocked for posting until separately warmed.

## Major Changes

### Local Web Cockpit

- The browser cockpit at `http://127.0.0.1:8765/app` is now the supported UI.
- The retired SwiftPM/macOS app path has been removed from the shipped repo.
- Review, campaign, readiness, settings, and advanced workbench routes are covered by smoke and rendered browser QA.
- Review cards now show publish slot/state when an approved kit has been scheduled.

### MiniMax/Hermes Review Factory

- The supported production rhythm targets 24 review kits per local day.
- Active campaign cadence is eight kits per campaign per day across YourRAGE, PlaqueBoyMax, and JasonTheWeen.
- Fresh clip sourcing starts at 24h, then expands through 48h, 72h, 4d, and 5d.
- Quota recovery can expand into older windows when fresh supply cannot satisfy the daily target.
- Hermes build intents now run ensemble caption retiming before reporting success.
- Hook quality is locally gated so generic or duplicated hooks do not become review kits.

### Approval Publish Queue

- Approval now creates or updates a publish package for the kit.
- Approval schedules one publish job into the next future slot.
- Scheduled jobs use eight local slots per day: `00:14`, `03:14`, `06:14`, `09:14`, `12:14`, `15:14`, `18:14`, and `21:14`.
- The scheduler only assigns future slots; it does not backfill missed times into the past.
- Future scheduled jobs are rebalanced to avoid adjacent same-streamer slots when another campaign is available.
- Existing approved-but-unslotted kits are self-healed by `script/publish_schedule_tick.py`.
- The Settings page can reschedule already-approved kits into fresh future `:14` slots when stale jobs are found.

### Hermes Operations

- `script/install_hermes_clip_ops.sh` now installs:
  - `clip-ops scheduler tick`
  - `clip-ops publish schedule tick`
  - `clip-ops job dispatcher`
- The publish schedule tick is a no-agent job that promotes due scheduled jobs into package checks or `publish_live` jobs depending on local auto-post readiness.
- The dispatcher continues to own deterministic local execution for render, retiming, source, and publish work.

### Buddy Install Rail

- `script/codex_buddy_bootstrap.sh` gives incoming Codex sessions one guided setup path for friend machines.
- The bootstrap verifies no-key source health before asking for local keys.
- `script/queue_buddy_campaign_kickoff.py` queues the first campaign refresh, source discovery, and one review-kit build per active campaign through the Hermes job ledger.
- `docs/codex-buddy-bootstrap.md` provides a pasteable prompt so friends only need to supply local MiniMax/Hermes, Twitch, Kick, Upload-Post, profile, and warm-up answers.

## Safety Gates

- Live Upload-Post requires:
  - approved non-demo review kit
  - local Upload-Post API key
  - exact local Upload-Post profile name
  - selected-platform account warm-up marked complete
  - provider mode set to `live`
  - local auto-post arming or final GUI confirmation creating a `publish_live` job
- Package-check mode validates request payloads without uploading.
- No payout submission, account connection, or account rebrand automation is introduced.
- Secrets, local databases, browser state, rendered media, and runtime artifacts stay out of GitHub.

## Upgrade Notes

- Existing installs should rerun `./script/install_hermes_clip_ops.sh` to create the new publish schedule tick.
- Operators can manually run `PYTHONPATH=backend backend/.venv/bin/python script/publish_schedule_tick.py --json` once after upgrade to backfill already-approved kits.
- Settings now send backend modes `dry_run` and `live`; older UI values such as `uploadpost_dry_run` are no longer used.
- Release metadata is bumped to `0.2.0`; backend API version is `2026-06-14-approval-slots-04`.

## Known Non-Goals

- v0.2.0 does not post from unwarmed platforms or alternate Upload-Post profiles. Operator-armed auto-posting is intentionally limited to approved, scheduled, live-ready jobs.
- v0.2.0 does not submit payouts or connect/rebrand accounts.
- v0.2.0 does not tag a GitHub release until the PR is merged into `main`.
